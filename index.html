Rust + WebAssembly
layout engine

Places word clouds and other visualizations off the main thread, scaling from 50 to 50 000+ words. Framework-agnostic — your code does the drawing.

~1 ms for 100 words (Fast) ~88 ms for 50 000 words ~120× faster than d3-cloud Web Worker ready No bundler needed

Live demo

Runs in-browser via WebAssembly — no server-side computation.

canvas.html
your-app.js 
// One async call. No bundler needed.
import { layout } from "./js/index.js";

const items = [
  { text: "WebAssembly", weight: 100 },
  { text: "Rust",        weight: 80  },
  { text: "layout",      weight: 40  },
];

const placements = await layout(items, {
  mode:        "pretty",  // "pretty" | "balanced" | "fast"
  width:       960,
  height:      600,
  minFontSize: 14,
  maxFontSize: 88,
  padding:     2,
});
// → [{ text, x, y, rotation, fontSize }, ...]
// x/y are bbox center — render with center anchoring.

const ctx = canvas.getContext("2d");
ctx.textAlign    = "center";
ctx.textBaseline = "middle";
for (const p of placements) {
  ctx.save();
  ctx.translate(p.x, p.y);
  if (p.rotation) ctx.rotate(p.rotation * Math.PI / 180);
  ctx.font = `${p.fontSize}px sans-serif`;
  ctx.fillText(p.text, 0, 0);
  ctx.restore();
}

idle
worker-app.js 
// Layout runs on a dedicated Worker.
// Concurrent calls are safe — request IDs are matched.
import { createLayoutClient } from "./js/client.js";

const client = createLayoutClient();

const placements = await client.layout(items, {
  mode:   "pretty",
  width:  960,
  height: 600,
});

// Render exactly as before — x/y are still bbox centers.
// Call client.terminate() when you're done with the worker.
client.terminate();

Treemap — same engine, different layout

The LayoutEngine trait is generic. Treemap is a fully-implemented strategy with its own output type — areas proportional to weight, no overlaps.

treemap.html
treemap-app.js 
import { treemap } from "./js/index.js";

const rects = await treemap(
  [
    { text: "Rust",   weight: 78 },
    { text: "wasm",   weight: 40 },
    { text: "layout", weight: 30 },
    /* … */
  ],
  { width: 960, height: 600 },
);
// → [{ text, value, x, y, width, height }, ...]
// x/y are top-left — render with fillRect.

const ctx = canvas.getContext("2d");
for (const r of rects) {
  ctx.fillRect(r.x, r.y, r.width, r.height);
}

Performance

Host CPU, scalar kernels, release build. Identical deterministic inputs for both engines (Zipf weights, seed 1). Full methodology in BENCHMARKS.md.

Fairness caveat: d3-cloud measures real glyph outlines; this engine uses approximate advance-width boxes — part of d3-cloud's cost is work this engine currently skips. Both single-threaded. Parallel rasterization (~2× for Fast at 50k) is on the roadmap.

How it works

You hand the engine a list of words; it hands back coordinates. The calculation (steps 1–3) runs entirely in WebAssembly; the drawing happens outside it, in whatever renderer you choose — Canvas, SVG, D3, React, or any charting tool that can place text.

1

Give it words and weights

The input is a list of { text, weight }. A word's weight decides how big it should be — the heaviest word becomes the largest, the rest scale down from there.

2

Turn each word into a shape

Every word is drawn once at its target size and traced into a compact black-and-white stencil of the pixels it covers. This is the only step that touches fonts, and the words are processed in parallel across CPU cores.

3

Pack the shapes without overlap

Each word spirals outward from the center looking for a gap. To check a spot, its stencil is compared against a single map of pixels already taken — a fast bitwise test that stays quick even at tens of thousands of words. A mode tunes this: Pretty for a tight spiral look, Fast for huge counts, Balanced in between.

4

Get back coordinates to draw

The result is { x, y, rotation, fontSize } for each placed word — position only, no pixels. You draw it however you like (Canvas, SVG, D3, React…). Run the whole thing in a Web Worker and the page never freezes, even for 50 000 words.
